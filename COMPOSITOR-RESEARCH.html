The Two-Condition Crash Model

Root Cause: Two conditions must be present simultaneously The crash loop requires BOTH a DCN pipeline stall AND GFX ring pressure from the compositor. Remove either condition and the system is stable.

Condition 1: DCN Pipeline Stall

optc31_disable_crtc REG_WAIT timeout during EFI-to-amdgpu handoff (~6s into boot). Caused by critically outdated DMCUB firmware (0.0.15.0).

Fix: Update DMCUB firmware to ≥ 0.0.224.0

Condition 2: GFX Ring Pressure

Compositor submits OpenGL commands to gfx_0.0.0 ring, which hangs on stalled DCN. MODE2 reset only resets GFX/SDMA — NOT DCN — creating an infinite loop.

Fix: Use a zero-ring-pressure compositor

GPU RING PRESSURE PATH (CRASHES): CPU (GL commands) → GFX Ring (gfx_0.0.0) → GPU Shaders → VRAM → KMS → DCN → Display ^^ TIMEOUT HERE ^^ PIXMAN / AccelMethod "none" PATH (SAFE): CPU (pixman/software) → System RAM (dumb buffer) → KMS page-flip → DCN → Display (GFX ring is NEVER touched)

Ring Architecture

RingEngineWhat Uses ItCrash Impact
gfx_0.0.0GFX (Graphics)OpenGL, GLES, Vulkan graphics queueCRASH TRIGGER

MODE2 Reset Limitation

Raphael APU uses MODE2 reset by default. MODE2 only resets GFX and SDMA via GCHUB. The DCN (Display Core Next) engine goes through DCHUB and is untouched by MODE2. This means a broken display pipeline persists through every GPU reset, causing the infinite crash loop.

amdgpu.reset_method=1 (MODE0) would reset ALL IP blocks including DCN/DCHUB, but is NOT SUPPORTED on Raphael APU — kernel 6.17 rejects it: “Specified reset method:1 isn’t supported, using AUTO instead.”

X11 Compositor Deep-Dive

AccelMethod: The Single Most Important Setting

The xf86-video-amdgpu DDX driver supports two acceleration methods:

AccelMethod2D RenderingGFX RingDRI3/VAAPIPage Flipping
"glamor" (default)GPU via OpenGL (radeonsi)All 2D ops use GFX ringFullYes
"none"CPU software renderingZERO GFX ring for 2DDRI3 works; DRI2 brokenDisabled
CONFIRMED: AccelMethod "none" eliminates the crash loop With AccelMethod "none" + XFCE + no NVIDIA, the system is STABLE: zero ring timeouts, zero GPU resets, reproducible across boots. 3D apps and VAAPI video decode still work via DRI3.

EXA is NOT available for amdgpu. It was deprecated and removed; only glamor and none exist. The modesetting DDX (generic) always uses glamor and has no "none" option.

Wayland Compositor Deep-Dive

The pixman Discovery

Key Finding: WLR_RENDERER=pixman eliminates ALL GFX ring submissions wlroots-based compositors (Sway, labwc, cage) support a pure CPU software renderer via the pixman library. With WLR_RENDERER=pixman, the entire compositing pipeline is CPU-only. The GFX ring is never touched, even during active window management.
PIXMAN PIPELINE (ZERO GFX RING): 1. Skip EGL/GL entirely — no eglCreateContext, no shader compilation 2. Create pixman image for each output (monitor) 3. pixman_image_composite32() for all compositing (SSE/AVX optimized) 4. Copy rendered image into DRM dumb buffer (linear framebuffer in system RAM) 5. KMS atomic commit to scan out the dumb buffer → Display controller operation (DCN path), NOT GFX ring

Weston (reference compositor) SAFE

Most mature pixman renderer of any Wayland compositor. weston --renderer=pixman. But limited as a desktop (no system tray, no task bar by default).

Hyprland / Wayfire — NOT RECOMMENDED HIGH

Hyprland: OpenGL ES 3.2 via Aquamarine. Continuous 3D animations. Does NOT support pixman. Known AMD iGPU crash issues (#5271).

Wayfire: 3D Compiz-like effects. Ubuntu 24.04 version (0.8.0) only supports GLES2, no pixman. Heavy GFX ring traffic.

Mutter (Wayland) — DO NOT USE CRITICAL

OpenGL only. No software rendering fallback on Wayland. LIBGL_ALWAYS_SOFTWARE=1 forces llvmpipe for clients but Mutter itself still creates hardware EGL context. Additional RT KMS thread SIGKILL bug (Mutter 46.x).

Vulkan Renderer: Does It Help?

No. wlroots Vulkan renderer uses the graphics queue (gfx_0.0.0), not compute. Only gamescope uses Vulkan async compute (comp_1.x.x ring), but it's not available on Ubuntu 24.04. Use pixman instead.

Client GPU Rendering with Pixman Compositor

With WLR_RENDERER=pixman, only the compositor's rendering is software. Client apps (Firefox, VS Code) can still use GPU acceleration via DRI. If a client gets a ring timeout, the GPU resets, and the client recreates its GL context — but the compositor survives because it never used the GFX ring. This breaks the infinite crash loop.

DMCUB firmware 0.0.15.0 is critically outdated The current firmware predates ALL known fixes by years. Target: 0.0.255.0 (conservative) or 0.1.40.0+ (latest stable). Avoid 0.1.14.0 (NixOS #418212 regression).

Key Firmware Facts

  • DMCUB firmware is NOT stored in persistent flash — loaded from filesystem on every boot
  • No risk of bricking hardware; rollback is trivial (restore old .bin.zst)
  • fwupd/LVFS does NOT deliver DMCUB firmware for integrated APUs
  • Ubuntu has CONFIG_FW_LOADER_COMPRESS_ZSTD=y — kernel loads .bin.zst FIRST. Manual .bin files are IGNORED if .bin.zst exists

Kernel Parameter Optimization

ppfeaturemask: Critical Finding

Running value 0xfff7bfff is WRONG — it disables the wrong bits 0xfff7bfff (current) disables OverDrive + GFX_DCS (irrelevant). 0xfffd7fff (correct) disables GFXOFF + stutter mode (critical for stability).

0xfff7bfff (WRONG — current)

Bits 14 & 19 cleared
Disables: PP_OVERDRIVE + PP_GFX_DCS
GFXOFF = ON, STUTTER = ON

0xfffd7fff (CORRECT)

Bits 15 & 17 cleared
Disables: PP_GFXOFF + PP_STUTTER_MODE
GFXOFF = OFF, STUTTER = OFF

DRI2 vs DRI3

DRI3 (default since Xorg ≥ 1.18.3) has lower ring pressure due to client-controlled buffer management and explicit fences. With AccelMethod "none", DRI version is mostly irrelevant for 2D but still applies to 3D apps.

GFXOFF: Belt-and-Suspenders

GFXOFF is controlled at two levels:

  1. BIOS: Advanced > AMD CBS > NBIO > SMU Common Options > GFXOFF = Disabled (confirmed)
  2. ppfeaturemask bit 15: amdgpu.ppfeaturemask=0xfffd7fff — currently set to WRONG value (0xfff7bfff) which does NOT disable GFXOFF

Both must be set. BIOS disables at firmware level; ppfeaturemask disables at driver level.

PCIe ASPM

The iGPU is on the internal fabric, not PCIe. ASPM settings primarily affect the RTX 4090. pcie_aspm=off prevents Xid 79 ("GPU has fallen off the bus"). CPU PCIE ASPM Mode should also be Disabled in BIOS.

Hardware vs Software Cursor

Hardware cursor uses the DCN cursor plane (no GFX ring). Software cursor is drawn by the compositor (GFX ring if GL). With AccelMethod "none", software cursor is CPU-rendered anyway, so it doesn't matter.

Architecture Goal

The Crash Loop MECHANISTIC ANALYSIS

Exact upstream match: drm/amd #5073 — Fence fallback timer expired on Raphael iGPU (OPEN)

Failure Sequence

6.1s — REG_WAIT timeout
optc31_disable_crtc — DCN register stall during EFI→amdgpu handoff. OTG_BUSY stays stuck because OTG_DISABLE_POINT_CNTL=2 waits for end-of-frame that never arrives.
8.0s — REG_WAIT timeout
optc1_wait_for_state — Cascading failure: OTG stopped, no VBLANK signals generated. This is NOT an independent bug.
18.5s — ring gfx_0.0.0 timeout
gnome-shell GFX ring hangs on corrupted display state. Compositor floods the ring with GL draw calls that stall on broken DCN.
19.0s — MODE2 GPU reset
Resets GFX + SDMA only. DCN is NOT touched. The broken display pipeline persists.
31.3s — ring gfx_0.0.0 timeout #2
gnome-shell immediately hits broken DCN again → another ring timeout → another MODE2 reset…
69.2s — ring gfx_0.0.0 timeout #3
GDM gives up → session-failed. Infinite crash loop.

Why MODE2 Reset Cannot Fix This

GPU IP Block Reset Architecture
GCHUB  ← GC (Graphics Compute) + SDMA   ← MODE2 resets THESE
MMHUB  ← VCN + JPEG + VPE               ← MODE1 resets THIS
DCHUB  ← DCN (Display Core Next)         ← NOT TOUCHED BY MODE2

MODE2 (default, reset_method=3) only resets GFX/SDMA. The DCN stays broken.
MODE0 (reset_method=1) does a full ASIC reset including DCN/DCHUB — can break the loop.
BACO (reset_method=4) not applicable to APUs (no separate power domain).
UPDATE: Testing confirmed MODE0 is NOT SUPPORTED on Raphael APU — kernel 6.17 rejects it.

Two Conditions Required for Crash

Condition 1: DCN Stall at Boot

optc31_disable_crtc timeout at ~6s — stalls DCN pipeline during BIOS→amdgpu handoff. Caused by outdated DMCUB firmware + missing kernel patches.

Fix: Update DMCUB firmware ≥ 0.0.224.0 + kernel ≥ 6.17 with DCN31 patches.

Condition 2: Compositor GFX Ring Pressure

GNOME/Mutter floods the GFX ring with OpenGL draw calls. When the ring hits the stalled DCN, it hangs → MODE2 reset → repeat.

Fix: Use XFCE with compositing OFF (XRender, zero GFX ring submissions).

Remove EITHER condition and the crash loop breaks. The winning configuration addresses both.

Mutter RT-Priority SIGKILL Bug

Mutter 46.x creates a real-time priority KMS page-flip thread. When amdgpu is slow (DCN latency), the thread exceeds its RT scheduling deadline → SIGKILL → crashes gnome-shell independently of ring timeouts.

# Workaround (applied in autoinstall-stable.yaml):
MUTTER_DEBUG_KMS_THREAD_TYPE=user   # normal-priority thread

The .bin vs .bin.zst Conflict

Ubuntu kernel prefers .bin.zst — manual .bin is IGNORED

Ubuntu 24.04 has CONFIG_FW_LOADER_COMPRESS_ZSTD=y. Firmware loader checks:

  1. {name}.zst ← loaded first (from Ubuntu package, outdated)
  2. {name}.xz
  3. {name} ← manually placed .bin is never reached

Fix: Compress new firmware to .bin.zst, remove bare .bin, rebuild initramfs.

Why Kernel 6.17 Still Crashed

The failures were firmware + config, not kernel
  1. DMCUB firmware still ancient (0.0.15.0) — kernel patches can't compensate for broken firmware state machine
  2. .bin.zst priority — manual firmware update was completely ignored by kernel loader
  3. Stale initramfs — not rebuilt after switching kernels; old firmware baked in
  4. Config conflicts — modprobe.d and GRUB were inconsistent between test iterations

Conclusion: Kernel 6.17 is viable once firmware and parameters are correct.

3-Layer GFXOFF Defense

Layer 1: BIOS

Advanced → AMD CBS → NBIO → SMU Common Options → GFXOFF = Disabled

Hardware-level disable. Firmware honors it before OS loads.

Layer 2: ppfeaturemask

amdgpu.ppfeaturemask=0xfffd7fff

Bit 15 clear = GFXOFF feature disabled in driver power management subsystem.

Layer 3: gfx_off

amdgpu.gfx_off=0

Direct driver module parameter. Explicit disable. GRUB-only on kernel 6.8 (error -22 in modprobe.d).

modprobe.d Configuration

amdgpu.conf (safe params only)
options amdgpu sg_display=0
options amdgpu ppfeaturemask=0xfffd7fff
options amdgpu gpu_recovery=1
options amdgpu reset_method=1  # NOT SUPPORTED on Raphael APU — remove this line
options amdgpu dc=1

Escalation Guidance

reset_method=1 (MODE0) is NOT SUPPORTED

UPDATE: Testing confirmed MODE0 (reset_method=1) is rejected by kernel 6.17 on Raphael APU. Remove it from GRUB and modprobe.d:

# Remove reset_method=1 from GRUB cmdline entirely
# Remove "options amdgpu reset_method=1" from modprobe.d/amdgpu.conf
# The default (-1, auto = MODE2 on Raphael) will be used
sudo update-initramfs -u -k all && sudo reboot
If dcdebugmask=0x10 still shows REG_WAIT

Escalate the debug mask:

  1. 0x10 — Disable PSR only (current)
  2. 0x18 — + Disable DCN clock gating
  3. 0x1C — + Disable stutter (display self-refresh)
